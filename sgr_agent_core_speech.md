## Блок 1. Что такое SGR Agent Core (0:05 - 0:15)

**[Слайд: SGR vs обычный агент]**

Давайте начнем с главного вопроса. В мире уже десятки фреймворков для агентов - LangChain, LlamaIndex, CrewAI. Зачем нужен еще один?

**[Слайд: Schema Guided Reasoning]**

SGR - это Schema Guided Reasoning. Это попытка сделать рассуждения агента детерминированными и интерпретируемыми. 

Чем отличается от обычного чат-бота с tool calling? В классическом агенте вы видите черный ящик: запрос ушел, какая-то магия произошла, ответ пришел. В SGR каждый шаг - это структурированный выбор, который можно валидировать. Мы явно описываем фазы: рассуждение, планирование, действия, ответ.

**[Слайд: минимальный код]**

Вот минимальный пример использования прямо из Python - буквально 20 строк. Создаем клиент OpenAI, конфигурацию, набор тулов и запускаем агента. Никакой магии - чистая конфигурация.

Главное преимущество? Стабильность и валидируемость результатов. Вы можете проследить, почему агент принял именно это решение.

## Блок 2. Архитектура API сервера (0:15 - 0:25)

**[Слайд: архитектура FastAPI]**

Теперь про серверную часть. SGR Agent Core предоставляет FastAPI сервер, который хранит агенты в памяти и управляет их жизненным циклом.

**[Слайд: жизненный цикл]**

Жизненный цикл простой: создание, выполнение, завершение по таймауту. Агент живет в памяти сервера, и у него есть TTL.

**[Слайд: stateless vs stateful]**

Два режима работы. Stateless - клиент шлет полный контекст каждый раз. Это как REST API - никакой памяти между запросами. Stateful - диалог по agent_id, история хранится на сервере. Удобно, когда хотите продолжить разговор без пересылки всей истории.

Важно: stateful-агент живет в памяти сервера и имеет таймаут. Не забывайте об этом в production.

**[Демо: curl запросы]**

Сейчас я покажу базовый стриминговый запрос и stateful-запрос. Обратите внимание на формат - он полностью совместим с OpenAI API. Это значит, что вы можете подключить наш сервер к Cursor, Claude Desktop, ChatGPT UI или любому другому клиенту.

## Блок 3. Конфигурация YAML и типы агентов (0:25 - 0:35)

**[Слайд: иерархия конфига]**

Ключевая философия SGR - конфигурация вместо кода. Вот иерархия: сначала дефолты в коде, потом переменные окружения, потом глобальный config.yaml, потом специфичный для агента agents.yaml.

**[Слайд: типы агентов]**

Три типа агентов. SGRAgent - базовый, с явными шагами рассуждения. ToolCallingAgent - классический tool calling для простых сценариев. SGRToolCallingAgent - гибрид, где SGR контролирует стратегию, но тулколлинг выполняет действия.

**[Слайд: пример config.yaml]**

Вот типичный конфиг. Блок llm с api_key, base_url, моделью. Блок execution с лимитами на уточнения и итерации. Блок tools с настройками конкретных тулов. И блок agents, где мы определяем конкретные агенты с их классами и наборами тулов.

Ключевая идея - минимальные переопределения. Базовый конфиг задает LLM и execution, а конкретный агент только выбирает тулы и класс. Не нужно дублировать настройки в каждом агенте.

## Блок 5. Практика: Deep Research (0:45 - 1:00)

**[Слайд: пайплайн Deep Research]**

Первая практика - Deep Research. Это агент, который делает глубокое исследование темы. Пайплайн такой: генерация плана, поиск в интернете, извлечение содержимого страниц, адаптация плана, финальный ответ.

**[Слайд: набор тулов]**

Набор тулов: GeneratePlan создает план, WebSearch ищет, ExtractPageContent парсит страницы, AdaptPlan корректирует план по ходу, FinalAnswer формирует результат.

**[Демо: запуск и запрос]**

Копируем конфиг для deep research, прописываем ключи Tavily для поиска и OpenAI для LLM. Запускаем. Теперь шлем запрос через curl - исследуй рынок RAG решений.

**[Слайд: логи и отчеты]**

Смотрим в логи. Видим каждый шаг: агент сгенерировал план, выполнил поиск, адаптировал план, сформировал ответ. Это не черный ящик - прозрачный пайплайн, поверх которого можно строить метрики.

Отчеты сохраняются в reports/. Логи в logs/.

## Блок 6. Практика: Файловый агент (1:00 - 1:40)

**[Слайд: переход к файловому агенту]**

Теперь главное - файловый агент. Это то, ради чего многие здесь собрались. Мы переходим от research-сценария к работе с локальными файлами.

**[Слайд: файловые тулы]**

Набор тулов для работы с файловой системой. GetSystemPathsTool - стандартные пути. ListDirectoryTool - листинг с рекурсией. ReadFileTool - чтение с диапазоном строк. SearchInFilesTool - grep-подобный поиск. FindFilesFastTool - поиск по паттерну, размеру, дате.

Важно: read-only по умолчанию. working_directory - это корень, за который агент не выйдет. Безопасность прежде всего.

**[Слайд: 4 режима работы]**

Четыре режима работы агента. HTTP сервер - интеграция с любыми OpenAI-совместимыми клиентами. Stateful - продолжение диалога по agent_id. Библиотека - встраивание в свой Python-код. ACP - интеграция с Obsidian, IDE и другими MCP-клиентами.

**[Демо: HTTP API с gpt-oss]**

Я покажу HTTP API с gpt-oss 120b - моделью, которую можно запускать локально. Конфиг почти такой же, только base_url указывает на ваш локальный сервер, а модель - gpt-oss:120b.

Запускаем сервер с файловым конфигом. Шлем запрос: найди в заметках все упоминания SGR за последний месяц. Агент ищет, читает файлы, отвечает.

**[Слайд: ACP и Obsidian]**

ACP - Agent Communication Protocol - это мост между SGR и Obsidian. sgracp запускает stdio-сервер, который Obsidian Agent Client понимает.

Минимальный блок acp в yaml - всего две строки: acp и имя агента. Устанавливаете плагин через BRAT, настраиваете Custom Agent - и говорите агенту "найди в моих заметках".

**[Демо: Obsidian -> sgracp -> vault]**

Смотрите схему: Obsidian отправляет запрос, Agent Client передает его sgracp, агент работает с вашим vault, возвращает результат. Все в рамках вашей системы, никаких облаков.

У вас 30 минут на эту часть. Задача - запустить файлового агента, сделать 2-3 запроса к своей папке с заметками. Кто освоится быстро - попробуйте подключить к Obsidian через ACP.

## Блок 7. Метрики, тесты, Langfuse (1:40 - 2:00)

**[Слайд: уровни тестирования]**

Мы научились запускать агентов. Теперь про качество. Уровни тестирования: unit для отдельного тула, интеграция с фикстурой каталога, E2E для полного агента.

**[Демо: пример теста]**

Вот пример unit-теста для ReadFileTool. Создаем временный файл, вызываем тул, проверяем, что содержимое верное. pytest + tmp_path фикстура. Просто, но эффективно.

Запускаем: pytest examples/sgr_file_agent/tests/. Видим, что тесты проходят.

**[Слайд: метрики качества]**

Какие метрики важны? Полнота ответа - нашел ли агент все, что просили. Число лишних вызовов тулов - эффективность рассуждений. Время выполнения - latency на итерацию. Регрессия при смене модели - сохраняется ли качество, если поменяли gpt-4o-mini на gpt-oss.

**[Слайд: Langfuse]**

Langfuse для трассировки. Позволяет видеть каждый вызов тулов, измерять задержки, сравнивать прогоны, отслеживать стоимость. Добавляете блок observability в конфиг - и получаете полную картину.

**[Демо: пример конфига с Langfuse]**

Вот как выглядит конфиг. Блок observability с langfuse, public_key, secret_key, host.

Совет: тестируйте не только "правильный" путь. Проверяйте, что агент graceful обрабатывает отсутствующие файлы, пустые директории, странные запросы. Это то, что отличает прототип от production.

У вас 15 минут. Напишите один unit-тест для любого тула или посмотрите существующие тесты в репозитории.
